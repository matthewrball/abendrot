Full-plan audit run via `/ccg`. Codex read the file in full (engineering/risk/release). Gemini's sandbox blocked the file path, so it audited from the detailed prompt (design / Liquid Glass / GTM) — its points still apply. ## Codex (engineering / correctness / release) — verdict: direction right, not yet release-safe
1. **Gamma verification must NOT default to screen capture.** Measuring post-compositor pixels → ScreenCaptureKit → Screen Recording permission, contradicting "no permission." Make gamma a per-device/OS capability-classified + optional/lab-verified layer; overlay always the default fallback. *(FIXED inline §6.1/§19; §21‑E1.)*
2. **Overlay ≠ true color control** — it's a UX fallback with limits (fullscreen Spaces, Mission Control, login/lock, protected/HDR/EDR video, screenshots, multi-Space ordering). Don't call it "hardware warmth"; spec limits + test matrix.
3. **DDC restore is under-specified/unsafe.** Crash handlers can't reliably do async DDC; a monitor can stay altered. Need: EDID native snapshot, per-display transaction queue, rate-limit/backoff, write-then-read verify, **emergency "Restore Displays" command**, launch-time stale-state recovery, user warning when hardware mode active, and a decision: **DDC opt-in per display until proven**.
4. **Stable identity needs more than `CGDisplayCreateUUIDFromDisplayID`.** Define a `DisplayIdentity` (cgUUID + current displayID + EDID vendor/product/serial + transport + IORegistry path + NSScreen transient). Debounce reconfiguration bursts on the main actor.
5. **Private-API mitigation needs a kill switch + release policy**, not just dlopen/dlsym: runtime feature flags, OS-build denylist, "private APIs disabled" fallback mode, and **signed/notarized private-API smoke tests from M0**.
6. **"Follow Night Shift" is oversold** — it's best-effort private state, not a schedule contract. Rename internally `SystemNightShiftStateFollower`; copy "follow system Night Shift *when available*"; add manual/approx-timezone fallback before requesting Location Services.
7. **Screenshot/recording auto-suspend not reliable for v1.0** — defer auto-detection; ship manual reveal + a "temporarily reveal during captures" shortcut.
8. **Per-website exclusions out of scope** for v1.0 (needs browser integration/Accessibility). v1.0 = **per-app exclusions only**; per-website → future.
9. **Sparkle key vs CI conflict** — can't be both keychain-only AND used by CI to sign appcasts. Pick ONE release authority: local keychain-only, OR GH Actions environment-protected secret with manual approval. Document rotation/revocation.
10. **Notarization tested earlier** — M0 signed+hardened+notarized smoke build; parse `notarytool log`; verify `spctl -a -vvv`.
11. **DMG needs a deterministic fallback** — two modes: `pretty-dmg` (branded Finder window, UI runner) + `plain-dmg` (deterministic `hdiutil` drag-to-Applications). Gate release on ≥1 notarized+stapled DMG. Mount in CI and verify layout, signature, quarantine first-launch, `/Applications` drag-install. (create-dmg AppleScript hangs headless — issue #154.)
12. **Homebrew cask release contract** — `livecheck`, `auto_updates true` (Sparkle), `zap` stanzas, SHA256, versioned release URLs; publish only AFTER Sparkle/appcast + DMG are coherent.
13. **CI "per 26.x point release" unrealistic on hosted runners** — split: hosted (lint/unit/archive/sign-dry-run) + self-hosted physical matrix (M5 Tahoe gamma-broken, M3/M4, Apple display, DDC monitor, HDMI/dock) + manual fresh-user Gatekeeper gate.
14. **Test plan needs failure-injection/persistence** — crash during DDC write, SIGKILL, wake while display service gone, hotplug during reveal-hold, lost key-up across Space switch, duplicate identical monitors, ColorSync profile changes, Night Shift/TrueTone/HDR on, competing apps (f.lux/Lunar/BetterDisplay/MonitorControl).
15. **One-push full 1.0 = wrong PUBLIC release strategy.** Keep the polished 1.0 ambition, but precede it with signed public betas: `0.1` overlay+hotkey+schedule+DMG+notarization → `0.2` DDC opt-in + restore tooling → `0.3` Sparkle → `1.0` branded launch after the hardware matrix passes.
- **Module split:** `WarmthCore` (pure math/schedule/state, no AppKit/IOKit) · `DisplayServices` (identity/hotplug/ColorSync/gamma) · `HardwareDDC` (private IOAVService behind protocols) · `OverlayRenderer` (AppKit/Metal, main-actor) · `NightShiftBridge` (private CBBlueLightClient, optional) · `AbendrotApp`. Keep `CInterop` thin; every private lookup → a typed capability result, not optionals sprinkled through the engine.
- **Codex top 5:** (1) drop default screen-capture gamma probe; (2) DDC opt-in until restore/recovery/EDID/verify/reset proven; (3) reframe overlay as documented-limitation fallback + expand testing; (4) private-API feature flags + OS denylists + signed smoke tests from M0; (5) staged notarized betas before the polished 1.0.

## Gemini (Liquid Glass / design / GTM)
1. **Liquid Glass (Tahoe-native, not 14/15 baseline):** add **specular tracking** (cursor-aware edge glint), **variable-thickness/lens blur** (tighter blur at edges = physical volume), a **material hierarchy** (clear transient glass for the popover; more-opaque "frosted ember" for the persistent/data-heavy Settings so text stays legible), and a **Reduce-Transparency fallback = ember-tinted SOLID** (not neutral grey — preserve the warm identity when opaque).
2. **Icon iteration = 3-3-1 strategy:** pure glyph / glass-pebble squircle / abstract orb. Build a **"vibrant template"** 18px menu-bar icon that subtly **glows amber when active**, tested against **desktop wallpaper tinting** (needs a faint outer glow or 0.5pt stroke so it never disappears on warm wallpapers).
3. **DMG = unboxing:** split-screen **cold→warm background** (the install window itself demos the product — drag the app from "cold/blue" to the "warm" Applications side); a higher-gloss DMG-internal app icon; and a **"move to /Applications" Liquid Glass HUD** (LetsMove/Sindre pattern) instead of a stock dialog, keeping immersion unbroken.
4. **GTM:** add an **"Audit the engine" code-snippet** on the landing page (show the real `WarmthKit` DDC write) to prove transparency to the NightOwl-burned crowd; **tailor the PH/Show HN LEAD to the reliability/designer hook** ("f.lux is broken on my M5", Reveal True Color) — the tech crowd is cynical about "circadian health" — while keeping health as the brand story; add a **v0.9 designer beta ~2 weeks pre-PH** to harvest real Liquid-Glass-UI screenshots for "social proof of beauty."
5. **UX feel:** spring-loaded **"lift the veil"** reveal (SwiftUI `.interactiveSpring`, not just a fade); **advanced mode = "liquid expansion"** of the popover (glass grows) for inline power rows, with the Settings window for deep config; tighten onboarding to **"3 clicks to warmth"** (permit notifications → set max warmth → confirm schedule).
- **Already in the plan (validated, not new):** Gemini's "show the warmth method (Hardware/Gamma/Overlay)" is already a core differentiator (§4.1) and "measured transparency" top-5 — good independent confirmation.
- **Gemini top 5:** specular tracking · interactive-spring reveal · split-screen DMG · measured/transparent warmth method (already done) · ember-tinted Reduce-Transparency fallback.
